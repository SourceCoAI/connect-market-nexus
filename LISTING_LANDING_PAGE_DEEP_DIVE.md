# Listing & Landing Page Deep Dive — 2026-03-05

## Executive Summary

This audit covers the entire pipeline from deal data to marketplace listing to landing page to connection request. After reviewing every file in the chain, I've identified **critical bugs**, **inconsistency sources**, **missing information patterns**, and **funnel leaks** that are causing the quality issues you're seeing.

---

## 1. THE FULL PIPELINE (How It Works Today)

```
Deal Data (listings table, is_internal_deal=true)
    |
    v
[1] deal-to-listing-anonymizer.ts — structural anonymization (title, description, hero)
    |
    v
[2] CreateListingFromDeal.tsx — pre-fills listing editor form
    |
    v
[3] generate-lead-memo edge function — AI generates content sections
    |
    v
[4] ImprovedListingEditor.tsx — admin reviews/edits
    |
    v
[5] use-robust-listing-creation.ts — saves to DB as internal draft
    |
    v
[6] publish-listing edge function — quality gates, then publishes
    |
    v
[7] ListingDetail.tsx (authenticated) / DealLandingPage (public) — buyer-facing
    |
    v
[8] ConnectionButton / DealRequestForm — connection request
```

There are also TWO SEPARATE teaser generation paths:
- **Path A**: `generate-lead-memo` with `memo_type: "anonymous_teaser"` — generates from RAW deal data
- **Path B**: `generate-teaser` — generates from a COMPLETED lead memo (memo-to-teaser transform)

These two paths use **completely different prompts, section structures, and validation rules**, which is a primary source of inconsistency.

## 2. CRITICAL BUGS FOUND

### BUG 1: Section Header Mismatch Between Two Teaser Generators

**`generate-lead-memo` (anonymous_teaser mode)** expects these sections:
- BUSINESS OVERVIEW
- DEAL SNAPSHOT
- KEY FACTS
- GROWTH CONTEXT
- OWNER OBJECTIVES

**`generate-teaser`** expects these sections:
- COMPANY OVERVIEW
- FINANCIAL SNAPSHOT
- SERVICES AND OPERATIONS
- OWNERSHIP AND TRANSACTION
- MANAGEMENT AND STAFFING
- KEY STRUCTURAL NOTES

**Impact**: When the `generate-teaser` function creates content, it uses the **full memo section headers** (COMPANY OVERVIEW, FINANCIAL SNAPSHOT, etc.). But the `buildHeroFromMemo()` function in `generate-lead-memo` looks for keys like `business_overview` and `deal_snapshot`. If content was generated by `generate-teaser`, the hero builder won't find the right sections, producing a blank or poor hero description.

**File**: `supabase/functions/generate-lead-memo/index.ts:469-475`

### BUG 2: Teaser Validation in `generate-teaser` Checks Wrong Headers

The `generate-teaser` function's `validateTeaser()` checks for `## COMPANY OVERVIEW` as required, but its system prompt tells the AI to use section headers matching the full lead memo format. Meanwhile, the sections that actually get displayed on the landing page (`ContentSections.tsx`) just render whatever `custom_sections` contains — so mismatched headers between the two generators lead to inconsistent display.

**File**: `supabase/functions/generate-teaser/index.ts:154-167`

### BUG 3: Description Overwrite Race Condition

In `CreateListingFromDeal.tsx`, there are two content sources competing:

1. **`anonymizeDealToListing()`** generates a description immediately from deal fields (line 86)
2. **`generate-lead-memo`** is called async and overwrites the description when it returns (line 179)

If the AI call fails silently or returns partial content, the description stays as the anonymizer's output — which uses banned marketing language like "strong financial profile" and "well-positioned for continued growth" (see `deal-to-listing-anonymizer.ts:380-381, 443-444`).

**File**: `src/lib/deal-to-listing-anonymizer.ts:380` — `"The business demonstrates a strong financial profile"`
**File**: `src/lib/deal-to-listing-anonymizer.ts:443-444` — `"The business is well-positioned for continued growth with several identified expansion levers."`

### BUG 4: Landing Page Displays `geographic_states` (Anonymity Risk)

`ContentSections.tsx:138-154` displays `deal.geographic_states` as tags on the landing page. The anonymous teaser prompt says to NEVER include state names — but the landing page pulls states directly from the DB field, bypassing the anonymization.

**File**: `src/pages/DealLandingPage/components/ContentSections.tsx:138-154`

### BUG 5: Hardcoded "2024 Revenue" Label

`MetricsStrip.tsx:26` hardcodes the label "2024 Revenue" — this is now 2026 and the label is stale. The listing detail page may have the same issue.

**File**: `src/pages/DealLandingPage/components/MetricsStrip.tsx:26`

### BUG 6: `buildDealDataForAnonymization` Missing Key Fields

In both `ContentSections.tsx` and `DealHero.tsx`, the `buildDealDataForAnonymization()` helper sets `main_contact_name`, `main_contact_email`, `main_contact_phone`, `address_city`, and `address_state` all to `null`. This means `stripIdentifyingInfo()` has NO identifying terms to strip — the anonymization on the public landing page is effectively a no-op for contact info that might be embedded in description text.

**File**: `src/pages/DealLandingPage/components/ContentSections.tsx:15-39`
**File**: `src/pages/DealLandingPage/components/DealHero.tsx:10-35`

### BUG 7: DealRequestForm Grammar Issue

`DealRequestForm.tsx:109` says "Get full access to detailed financials and business metrics to {dealTitle}" — should be "**for** {dealTitle}" not "to".

**File**: `src/pages/DealLandingPage/components/DealRequestForm.tsx:109`

## 3. ALL AI PROMPTS (Complete Text)

### PROMPT 1: Full Lead Memo (generate-lead-memo, full_memo mode)

**Location**: `supabase/functions/generate-lead-memo/index.ts:1044-1116`

**System Prompt** (abbreviated key points):
```
You are a senior analyst at a tech-enabled investment bank writing an internal
lead memo. Your audience: partners and deal team members who evaluate opportunities.

CORE RULES:
1. ONLY STATED FACTS — every sentence must trace to source data
2. OMIT, DON'T APOLOGIZE — never write "not provided"
3. NO CHARACTERIZATION — no evaluative adjectives
4. NO COMPARISONS — no benchmarks unless in source data
5. MATCH DATA DENSITY — thin data = short memo (300-500 words), normal = 600-900

SOURCE PRIORITY:
1. Financial statements / tax returns
2. Call transcripts (most recent first)
3. General notes and manual entries
4. Enrichment data

SECTIONS (in order):
- COMPANY OVERVIEW (prose, 3-5 sentences)
- FINANCIAL SNAPSHOT (labeled lines, not tables)
- SERVICES AND OPERATIONS (bullets)
- OWNERSHIP AND TRANSACTION (bullets)
- MANAGEMENT AND STAFFING (bullets)
- KEY STRUCTURAL NOTES (only if structural complexity)

BANNED WORDS: 88+ words including strong, robust, impressive, attractive,
compelling, well-positioned, significant, poised for growth, etc.
```

**User Prompt**:
```
Generate a lead memo from the following data.

IMPORTANT: Transcripts may include SourceCo associates and the business owner.
Extract only facts about the target company stated or confirmed by the owner.

=== CALL TRANSCRIPTS (primary source) === {transcripts}
=== ENRICHMENT DATA (secondary source) === {enrichment}
=== MANUAL DATA ENTRIES & NOTES === {manual}
=== VALUATION CALCULATOR DATA === {valuation}

Return as markdown with ## headers. Headers must exactly match: COMPANY OVERVIEW,
FINANCIAL SNAPSHOT, SERVICES AND OPERATIONS, OWNERSHIP AND TRANSACTION,
MANAGEMENT AND STAFFING, KEY STRUCTURAL NOTES.
```

## 4. INCONSISTENCY SOURCES (Root Causes)

### 4.1 Two Competing Content Pipelines

| Aspect | `generate-lead-memo` (anonymous_teaser) | `generate-teaser` |
|---|---|---|
| Input | Raw deal data (transcripts, enrichment, etc.) | Completed lead memo text |
| Section Headers | BUSINESS OVERVIEW, DEAL SNAPSHOT, KEY FACTS, GROWTH CONTEXT, OWNER OBJECTIVES | COMPANY OVERVIEW, FINANCIAL SNAPSHOT, SERVICES AND OPERATIONS, OWNERSHIP AND TRANSACTION, MANAGEMENT AND STAFFING, KEY STRUCTURAL NOTES |
| Word Limit | 300-600 | 300-600 |
| Validation | Section headers, word count, banned words | Anonymity breach checks, section headers, word count, banned words |
| Anonymity Checks | Prompt-only (no code validation) | Code-validated (company name, owner name, cities, states, employee names) |

**Impact**: Depending on which path was used, the listing gets completely different section structures, different levels of anonymity validation, and different content styles.

### 4.2 Triple-Layer Content Generation

Content can come from THREE sources:
1. **`deal-to-listing-anonymizer.ts`** — deterministic code, uses banned marketing words
2. **`generate-lead-memo` anonymous_teaser** — AI from raw data, newer section structure
3. **`generate-teaser`** — AI from completed memo, old section structure

These three sources produce different quality levels, different formatting, and different section structures. When AI generation fails, the fallback (source 1) uses marketing language that the AI prompts explicitly ban.

### 4.3 Missing Data Cascades

The pipeline doesn't validate data completeness BEFORE generation:
- If `transcripts` are empty, the AI gets almost nothing to work with
- If `enrichment` is empty, the description is thin
- No pre-flight check warns admins "this deal has insufficient data for a quality listing"
- The result: some listings are rich and detailed, others are skeleton-thin

### 4.4 Hero Description Built Differently in Two Places

1. **`deal-to-listing-anonymizer.ts:487-572`** — builds hero from raw deal fields
2. **`generate-lead-memo/index.ts:468-535`** — builds hero from AI-generated memo sections

The anonymizer builds a hero at listing creation time. Then if the AI teaser is regenerated later, `buildHeroFromMemo()` overwrites it — but only if a marketplace child listing exists. This means the hero can be stale or mismatched.

## 5. MARKETPLACE FUNNEL ANALYSIS

### Funnel: Landing Page -> Connection Request

```
Landing Page (DealLandingPage)
  |-- DealHero (title, hero_description, location)
  |-- MetricsStrip (revenue, EBITDA, team size, margin)
  |-- ContentSections (description + custom_sections)
  |-- DealRequestForm (name, email, company, phone, role, message)
  |-- DealSidebar (executive summary PDF, advisor info)
  |-- RelatedDeals (similar listings)
  |-- EmailCapture (non-submitters)
```

### Funnel Leaks Identified

**Leak 1: No CTA Above the Fold**
The `DealRequestForm` is buried at the bottom of the left column. There's no sticky CTA, no inline "Request Access" button near the hero. The sidebar has no request button on desktop — it only shows executive summary PDF and advisor info.

**Leak 2: Content Quality Variance**
When AI generation fails and the deterministic anonymizer's output is used, the content quality drops dramatically. Marketing buzzwords ("strong financial profile", "well-positioned for growth") make the listing look generic and untrustworthy to sophisticated PE buyers.

**Leak 3: Missing Information on Landing Page**
The `buildDealDataForAnonymization()` in `ContentSections.tsx` sets most identifying fields to null, which means the anonymization pass is largely a no-op. This means either:
- Content already came in pre-anonymized (good) — but there's no guarantee
- Contact info embedded in description text doesn't get stripped (bad)

**Leak 4: Form Requires Phone Number**
`DealRequestForm.tsx:157` marks phone as required. PE associates filling out a quick form may bounce on this requirement. Phone should be optional — email is the primary contact method.

**Leak 5: No Social Proof or Urgency**
The landing page has no indicators of deal popularity (number of requests, time listed), no urgency signals, and no social proof. Compare to typical investment bank teasers that include "Process Update" status.

**Leak 6: Post-Form Success State Is Weak**
After submission, the user sees a "Join the Marketplace" nudge. There's no confirmation email, no timeline expectation ("typically responds within 48 hours"), and the "Schedule a Call" button links to a signup page, not an actual scheduling tool.

### Funnel: Marketplace -> Listing Detail -> Connection Request

```
Marketplace (authenticated)
  |-- ListingCard (title, financials, badges)
      |
      v
ListingDetail
  |-- ListingHeader (title, categories, location)
  |-- EnhancedFinancialGrid (revenue, EBITDA, team, margin)
  |-- BusinessDetailsGrid (states, services, locations)
  |-- EditableDescription (main description + HTML)
  |-- CustomSections (lead memo content)
  |-- ConnectionButton (primary CTA)
  |-- InvestmentFitScore (match analysis)
  |-- DealAdvisorCard
  |-- SimilarListingsCarousel
  |-- BuyerDataRoom (after connection approved)
```

This funnel is more complete but has its own issues:

**Leak 7: Multi-Gate System Is Confusing**
Users must pass: email verification -> profile completeness -> fee agreement -> NDA -> connection request. This is a LOT of friction. Each gate is a potential drop-off point with no progress indicator showing "you're 3 of 4 steps from access."

**Leak 8: ConnectionRequestDialog Discouraging Copy**
`ConnectionRequestDialog.tsx` shows "40-50 requests per deal, only 1-3 are introduced." This is meant to encourage quality messages but may discourage buyers from even trying.

## 6. RECOMMENDATIONS

### Immediate Fixes (Bugs)

1. **Standardize section headers** across both teaser generators — pick ONE set of headers and use it everywhere
2. **Fix banned words in anonymizer fallback** — remove "strong financial profile", "well-positioned for continued growth" from `deal-to-listing-anonymizer.ts`
3. **Fix geographic_states display on landing page** — either convert to regions or hide on the public landing page
4. **Fix "2024 Revenue" hardcoded label** — use `deal.revenue_metric_subtitle` or make it dynamic
5. **Fix DealRequestForm grammar** — "to" -> "for"
6. **Fix buildDealDataForAnonymization** — pass through all available identifying fields so the anonymization actually works

### Content Quality Improvements

7. **Add data completeness pre-check** before AI generation — warn admin if transcripts/enrichment are missing, and show a quality score
8. **Eliminate the deterministic anonymizer fallback for description/hero** — if AI fails, show an empty editor with guidance instead of low-quality generated text
9. **Unify the two teaser generation paths** — there should be ONE way to generate an anonymous teaser, not two with different formats
10. **Add a preview mode** — let admins see exactly how the listing/landing page will look before publishing

### Funnel Improvements

11. **Add a sticky CTA on landing page** — "Request Full Details" button that scrolls to or opens the form
12. **Make phone optional** on DealRequestForm
13. **Add social proof** — "X buyers have requested this deal" or "Listed 3 weeks ago"
14. **Simplify the multi-gate system** — combine profile completion + NDA into a single onboarding flow with progress indicator
15. **Add timeline expectation** to post-submission state — "Our team typically responds within 2 business days"

### Prompt Improvements

16. **Add structured output format** to AI prompts — use JSON schema or explicit delimiters instead of relying on markdown parsing (which is fragile)
17. **Add a "minimum data quality" guard** — if the deal has < 3 data fields populated, refuse to generate and tell admin to add more data
18. **Add validation for financial consistency** — if AI-generated revenue doesn't match deal.revenue within 20%, flag it
